# Structured Ideation

## Core Principle

Ideas are cheap. Validated ideas are expensive. This skill bridges the gap — structured questioning that kills bad ideas fast and strengthens good ones.

## Six Forcing Questions

Before any implementation commitment, answer all six. If you can't answer one clearly, the idea isn't ready.

### 1. Demand Reality
"Who specifically needs this, and how do you know?"
- Name a real person, company, or concrete persona — not "users" or "developers"
- Evidence: did they tell you? Did you observe it? Are you guessing?
- If the answer is "I think people would want..." — STOP. Go find evidence first.

### 2. Status Quo
"What do they do today without this?"
- Every problem has a current workaround — understand it
- If the workaround is "nothing, they just suffer" — high opportunity
- If the workaround is "they use X and it's fine" — you need to be 10x better

### 3. Desperate Specificity
"How desperate are they? Would they use a half-broken version?"
- If yes — real demand. Build the minimum and ship.
- If "only if it's polished" — weak demand. They're fine without it.
- The best products solve problems people are already hacking around

### 4. Narrowest Wedge
"What is the smallest version that delivers value?"
- Not MVP (which has become "everything minus polish") — the NARROWEST wedge
- One user, one use case, one workflow. That's it.
- If you can't narrow it to one workflow, the idea is too broad

### 5. Surprise Factor
"What surprised you during research?"
- Surprises are where insight lives — they mean your mental model was wrong
- If nothing surprised you, you didn't research deeply enough
- The best product ideas come from "I expected X but found Y"

### 6. Time Vector
"Does this problem get better or worse over time?"
- Getting worse → urgent, growing market, build now
- Getting better → someone else is solving it, differentiate or skip
- Stable → lifestyle business, not a rocket ship (fine if that's your goal)

## Modes

### Startup Mode (validated demand exists)
Focus on questions 3, 4, 6. Demand is proven — now find the narrowest wedge and ship fast. Skip extensive brainstorming.

### Builder Mode (exploring an idea)
Focus on questions 1, 2, 5. Demand is unproven — prioritize evidence gathering over building. The goal is to kill the idea or validate it, not to design the solution.

### Growth Mode (extending existing product)
Focus on questions 2, 4, 6. Product exists — find the next wedge. What's the status quo for adjacent use cases? What's the smallest extension that opens a new segment?

## Design Lineage

Track how ideas evolve across sessions:
- When an idea is refined, note what changed and why
- Maintain a revision chain: "v1: full platform → v2: single workflow → v3: CLI tool for one task"
- This prevents re-litigating settled decisions and shows the reasoning path

## When to Use

- Before writing any product spec or PRD
- When someone says "we should build X" — run the six questions first
- Quarterly planning, roadmap reviews, feature prioritization
- Hackathon ideation, side project evaluation
- Not just software — works for content, services, processes

## Anti-Patterns

- **Building before asking**: writing code before answering question 1
- **Idea attachment**: falling in love with the solution before validating the problem
- **Breadth over depth**: brainstorming 20 ideas instead of deeply validating 1
- **Skipping the status quo**: not understanding what people do today leads to building something nobody switches to
- **Consensus-seeking**: good ideas often sound wrong to most people initially. Conviction from evidence beats consensus from comfort.
